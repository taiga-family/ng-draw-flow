The optional @ng-draw-flow/layouts package calculates node positions without coupling layout algorithms to the core editor. The first bundled implementation produces strict trees and uses d3-hierarchy internally; applications configure only the supported strict-tree options.

Installation

This guide assumes that @ng-draw-flow/core is already installed. Add the optional layouts package with the same compatible release line.

Built-in Strict Tree

DfAutoLayoutService uses the bundled strict-tree implementation by default. Applications do not instantiate or call D3. Inject the service and request a layout when the editor is initialized or its model changes:

Without configuration, the tree flows left to right, uses a fixed 180 x 64px node size, an 80px level gap and a 32px sibling gap, and preserves the current root position. Provide only the options that differ for a graph. A component-level provider scopes both configuration and the auto-layout service to that editor:

The engine supports four explicit directions:

  • LeftToRight
  • RightToLeft
  • TopToBottom
  • BottomToTop

levelGap controls spacing between generations and siblingGap controls spacing within a generation. preserveRootPosition defaults to true; disable it to position the root at origin instead. Use rootId to require a particular node as the root; it must still have no incoming connection and be the graph's only root.

Node Sizing

Use DfNodeSizingStrategy.Fixed when every node has the same dimensions. Its size is used directly during every calculation and does not require DOM measurements.

Use DfNodeSizingStrategy.Measured for nodes whose dimensions depend on content, templates or expanded sections. Core then publishes actual wrapper dimensions through its existing resize observation. Layouts performs an initial calculation with the required fallback and automatically reapplies the layout after all current nodes are measured. Later size changes are coalesced per animation frame and also trigger a recalculation.

Measured dimensions are runtime view state and are not written to DfDataModel . The fallback is also used before rendering and whenever a newly added node has not been measured yet.

Connectors

Layouts calculate node positions but do not synthesize connection points. Core measures the user-provided df-input and df-output components, preserving full freedom over their number and placement. Every connector id referenced by a connection must therefore be rendered by its node.

When regular connection creation is disabled, input connectors become invisible but remain available as target anchors. Action outputs stay visible and each connector can create an independent tree branch.

Apply and Reapply the Layout

DfAutoLayoutService reads the current model from NgDrawFlowStoreService , runs the configured engine, and writes the returned model back to the editor. Calling apply() does not return a Promise or require a subscription. Its running , result and error signals expose the latest operation state.

After an interactive mutation, pass anchorNodeId together with the new model to keep that node at its current canvas position without waiting for form synchronization. The service translates the complete result while preserving relative positions, so a repeatedly activated action output stays under the pointer. Selection is also retained across model updates while the selected node id or selected connection endpoints still exist.

Layouting does not change editor interaction options. Keep nodesDraggable enabled to arrange an ordinary graph once and then let users adjust it, or disable dragging when every graph mutation must remain under automatic layout control.

Smooth Position Changes

Node animation belongs to core and is opt-in, so it can also be used when positions come from another layout package or from application code. A duration of 0 keeps the existing immediate behavior.

Strict Tree Requirements

The built-in strict-tree layout accepts exactly one root, unique node ids, existing connection endpoints, one parent per child, and a graph where every node is reachable from the root. Validation failures are published to the error signal as DfTreeLayoutError instances.

The bundled engine is intentionally a strict-tree implementation. DAGs, cyclic graphs and nodes with multiple parents are not currently supported by the layouts package.

Dynamic Node Creation

A custom node can render one or more action outputs that create children. Each output controls its own id, content, constraints and application data; activating it changes the model and then reapplies the layout. When children use multiple outputs of the same node, assign each output a unique layoutOrder . Horizontal trees interpret it from top to bottom and vertical trees from left to right. Children created from the same output retain their model order. See the connector documentation for the complete action-output and custom-content API.
